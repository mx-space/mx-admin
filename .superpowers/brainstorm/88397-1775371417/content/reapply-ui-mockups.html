`re-apply` UI/UX 方向

这次不是文字方案，而是把它放回真实聊天上下文里看。重点看三件事：按钮放在哪、整组和单条怎么协同、冲突时用户有没有“先看再执行”的安全感。

Option A
Fastest
Hover 直接重放
把 `Re-apply` 做成 hover action，单条和整组都能在原位直接点。

优点是顺手，缺点是整组重放和单条重放都像“危险快捷键”。
Option B
Recommended
加到 Re-apply Tray
点击后先进入一个轻量 tray，用户可以混合单条、整组、diff batch，再统一执行。

Option C
Most agent-like
生成新的 Replay Bubble
点 `Re-apply` 以后，不直接改文档，而是在消息流里追加一条 “Replay request” 并重新运行。
Agent Chat

历史更完整，但 UI 上会更像“再发起一个 agent 任务”，不够像局部编辑动作。

我现在的推荐仍然是 B：入口可以保持轻量，但安全检查、冲突解释和三层统一抽象都会明显更顺。看完后告诉我你更接近哪张，或者直接说你想把哪两张混起来。
